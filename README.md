# Promt Drive_Record

ฉันกำลังจะพัฒนาเว็บแอปชื่อ Drive_Record ซึ่งเป็นระบบบันทึกรายรับ รายจ่าย และสรุปผลการวิ่งงานสำหรับผู้ขับรถรับส่ง เช่น Grab, Bolt, LINE MAN และแพลตฟอร์มอื่น ๆ โดยมี Roadmap ดังนี้

## Roadmap

Phase 1: ออกแบบ UX/UI และกำหนด Requirements

Phase 2: พัฒนา Frontend

Phase 3: พัฒนา Backend และ Database

Phase 4: ทำให้เป็น Progressive Web App (PWA)

---

ฉันต้องการให้คุณทำหน้าที่เป็น Senior Full Stack Developer, Software Architect และ Technical Mentor ช่วยวางแผนการพัฒนาโปรเจกต์นี้อย่างละเอียด โดยไม่ยึดติดกับเทคโนโลยีใดเป็นพิเศษ แต่ให้เลือกสิ่งที่เหมาะสมที่สุดสำหรับโปรเจกต์ พร้อมอธิบายเหตุผลของการเลือก

# 🚀 สรุป Tech Stack ที่แนะนำสำหรับปี 2026 (สำหรับ Solo Developer)

ในปี 2026 เทรนด์การพัฒนาเว็บเน้นไปที่ Full-Stack Frameworks และ BaaS (Backend as a Service) ที่ช่วยให้นักพัฒนาคนเดียวสามารถทำงานสเกลใหญ่ได้ ผมขอแนะนำ Stack ที่ "สมดุลที่สุด" ระหว่างความง่าย ประสิทธิภาพ และการต่อยอดดังนี้ครับ

## Core Framework: Next.js (React) + TypeScript

เหตุผล: เป็นมาตรฐานอุตสาหกรรม มีระบบ Routing ที่ดีเยี่ยม (App Router) รองรับการทำ PWA ได้ง่ายผ่านไลบรารีเสริม และมี Community ใหญ่ที่สุด การใช้ TypeScript จะช่วยลดบั๊ก (Typo/Type Mismatch) ได้มหาศาล ซึ่งสำคัญมากเวลาทำโปรเจกต์คนเดียว

## Styling & UI Components: Tailwind CSS + Shadcn UI

เหตุผล: Tailwind ทำให้เขียน CSS ได้เร็วโดยไม่ต้องสลับไฟล์ ส่วน Shadcn UI ให้ Component ที่สวยงาม รองรับ Accessibility (a11y) และสามารถปรับแต่งได้เต็มที่ ไม่เหมือน UI Library ยุคเก่าที่แก้ยาก

## Backend, Database & Auth: Supabase (PostgreSQL)

เหตุผล: สำคัญมากสำหรับ Solo Dev! Supabase คือ BaaS ที่ให้ฐานข้อมูล PostgreSQL, ระบบ Authentication, และ API สำเร็จรูปมาให้เลย คุณแทบไม่ต้องเขียน Backend Boilerplate เอง ทำให้ประหยัดเวลาไปได้กว่า 40% ของโปรเจกต์ และสเกลได้จริง

## State Management: Zustand

เหตุผล: จัดการ State (เช่น ข้อมูล User ที่ล็อกอิน) ได้ง่ายและเบากว่า Redux มาก

## Deployment: Vercel

เหตุผล: เกิดมาเพื่อ Next.js แค่ผูก Git Repository แล้วกด Deploy ระบบจะจัดการ CI/CD ให้ทั้งหมด ฟรีและเร็ว

## 4. ลำดับขั้นตอนการทำงาน (Step-by-step)

- Requirement Gathering: ลิสต์ฟีเจอร์ที่ต้องมี เช่น ระบบล็อกอิน, ฟอร์มบันทึกรายรับ-รายจ่าย, หน้า Dashboard สรุปยอด

- User Flow: วาดแผนผังว่าผู้ใช้เปิดแอปมาแล้วจะกดไปหน้าไหนบ้าง (เช่น หน้าแรก -> กดปุ่ม + -> กรอกตัวเลข -> บันทึก -> กลับหน้าแรก)

- Low-Fidelity Wireframe: ร่างหน้าจอคร่าวๆ เป็นกล่องๆ เพื่อดูตำแหน่งจัดวาง (เน้น Mobile Screen)

- High-Fidelity Design: ใส่สี จัด Typography (ขนาดตัวอักษร) และใช้ Component (ปุ่ม, Input)

- Prototyping: เชื่อมหน้าจอต่างๆ ใน Figma ให้กดทดสอบได้เหมือนแอปจริง

## 9. Best Practices ที่ควรใช้

- Thumb Zone Design: ปุ่มที่สำคัญที่สุด (เช่น ปุ่ม "บันทึก") ควรอยู่ด้านล่างขวา หรือตรงกลางล่าง ที่นิ้วโป้งเอื้อมถึงง่ายสุดเมื่อถือมือถือมือเดียว

- High Contrast & Dark Mode: คนขับรถทำงานกลางแดดและตอนกลางคืน หน้าจอต้องสู้แสงได้ และมี Dark mode ถนอมสายตา

- Big Tap Targets: ปุ่มต้องใหญ่พอ (อย่างน้อย 44x44 pixels) เวลารถสั่นจะได้กดไม่พลาด

## 10. ข้อผิดพลาดที่มือใหม่มักพบ พร้อมวิธีหลีกเลี่ยง

### ข้อผิดพลาด

ออกแบบเผื่อหน้าจอ Desktop ก่อน

### วิธีแก้

บังคับตัวเองให้ออกแบบหน้าจอ Mobile ให้เสร็จสมบูรณ์ก่อน 100% แล้วค่อยขยายเป็น Tablet/Desktop (Mobile-first อย่างแท้จริง)

### ข้อผิดพลาด

ใส่ข้อมูลทุกอย่างลงไปในหน้าเดียวจนรก

### วิธีแก้

ใช้หลัก Progressive Disclosure (แสดงเฉพาะข้อมูลที่สำคัญที่สุดก่อน ถ้าอยากดูรายละเอียดลึกๆ ค่อยกดเข้าไปดู)

# 💻 Phase 2: พัฒนา Frontend (Building the Face of Drive_Record)

## 1. เป้าหมายของ Phase นี้

เปลี่ยนดีไซน์จาก Figma ให้กลายเป็น Web Application ที่ใช้งานได้จริงบนเบราว์เซอร์

สร้างโครงสร้างโค้ดที่รองรับการขยายตัว (Scalable)

ทำให้แอปตอบสนองรวดเร็ว (Performance) และลื่นไหล (Smooth UI/UX)

---

## 2. เทคโนโลยีและเครื่องมือที่เลือกใช้ (The 2026 Choice)

### Next.js 15+ (App Router)

สุดยอด React Framework ที่จัดการ Routing, SEO และ Performance ให้คุณอัตโนมัติ

### TypeScript

บังคับให้คุณนิยามประเภทข้อมูล (เช่น รายได้ต้องเป็น Number) ช่วยป้องกัน Error ตั้งแต่ตอนเขียน

### Tailwind CSS

เขียน CSS ในตัว HTML เลย ทำให้จัด Layout ได้เร็วมาก และรักษา UI สม่ำเสมอ

### Shadcn UI

ชุด Component (ปุ่ม, ฟอร์ม, โมดัล) ที่สวยระดับพรีเมียม แต่คุณเป็นเจ้าของโค้ด 100% ปรับแต่งได้ทุกจุด

### Zod + React Hook Form

สำหรับจัดการ Form การบันทึกข้อมูลที่ซับซ้อนให้ง่ายและมีการตรวจสอบข้อมูล (Validation) ที่แม่นยำ

---

## 3. ตัวเลือกอื่น (Alternatives)

### Vite + React

(ข้อดี: เบามาก / ข้อเสีย: ต้องตั้งค่า Routing และ SEO เองทั้งหมด ซึ่งจะเหนื่อยกว่าในระยะยาว)

### Flutter for Web

(ข้อดี: เขียนครั้งเดียวได้ App จริงๆ / ข้อเสีย: ประสิทธิภาพบนเว็บยังไม่สู้ Next.js และการทำ SEO/Accessibility ยากกว่า)

ทำไมจึงเลือก Next.js: เพราะ Next.js มีทุกอย่างที่ PWA ต้องการ และในอนาคตถ้าคุณอยากทำ Server-Side Rendering (SSR) เพื่อความเร็ว Next.js ก็ทำได้ทันที

## 4. ลำดับขั้นตอนการทำงาน (Step-by-step)

- Project Initialization: ติดตั้ง Next.js พร้อม TypeScript และ Tailwind CSS

- Global Styles & Theme: ตั้งค่าสีตามดีไซน์ Figma (เช่น สีเขียว Grab, สีเหลือง LINE MAN)

- Core Components: สร้างปุ่ม, Input, และ Layout หลัก (Navbar/Bottom Navigation)

- Feature Pages: สร้างหน้าจอหลัก (Home, Add Record, History, Summary)

- State Management: ตั้งค่า Zustand เพื่อเก็บข้อมูลชั่วคราว (เช่น ข้อมูลที่ผู้ใช้กรอกค้างไว้)

- Responsive Polishing: ตรวจสอบความเรียบร้อยบนหน้าจอมือถือทุกขนาด

### หลังจบ

## 6. โครงสร้างโปรเจกต์ที่แนะนำ (Clean Architecture)

```
src/
 ├── app/            # Next.js Pages (Routes)
 ├── components/     # UI Components (Reusable)
 │    ├── ui/        # Shadcn UI (Low-level)
 │    └── shared/    # Header, Navigation, TransactionCard
 ├── hooks/          # Custom React Hooks (Logic แยกจาก UI)
 ├── lib/            # Utilities (Formatter, Supabase Client)
 ├── types/          # TypeScript Interfaces/Definitions
 └── store/          # Zustand State Management
```

## 9. Best Practices ที่ควรใช้

- Atomic Components: แยกส่วนประกอบเล็กๆ ออกมา เพื่อนำไปใช้ซ้ำ (เช่น ปุ่มบันทึกที่ใช้ทั้งหน้า Grab และ Bolt)

- Fast Load: ใช้ next/image และการทำ Code Splitting เพื่อให้แอปโหลดเสร็จภายใน 1-2 วินาที

# 🧠 Phase 3: พัฒนา Backend และ Database (The Brain & Memory)

## 1. เป้าหมายของ Phase นี้

สร้างฐานข้อมูลเพื่อเก็บข้อมูลผู้ใช้และรายการวิ่งงานอย่างปลอดภัย

สร้างระบบยืนยันตัวตน (Authentication) เช่น ล็อกอินด้วย Google หรือ Email

เปลี่ยนจากข้อมูลจำลอง (Mock Data) มาเป็นการดึง/บันทึกข้อมูลจริง

ทำระบบคำนวณสรุปยอด (รายวัน/รายเดือน) อย่างมีประสิทธิภาพ

---

## 2. เทคโนโลยีและเครื่องมือที่ควรใช้ (The 2026 Choice)

### Supabase (BaaS - Backend as a Service)

* เหตุผล: เป็นระบบจัดการฐานข้อมูล (PostgreSQL) และ Authentication ที่มีทุกอย่างพร้อมใช้งาน คุณไม่ต้องเช่าเซิร์ฟเวอร์แยก ไม่ต้องเขียนระบบล็อกอินเอง ประหยัดเวลาการทำ Backend ไปได้เกือบครึ่ง

### Next.js Server Actions

* เหตุผล: ใน Next.js 15+ คุณสามารถเขียนฟังก์ชันเชื่อมต่อ Database ได้โดยตรงจากฝั่งเซิร์ฟเวอร์ โดยไม่ต้องสร้าง API Route (REST API) ให้วุ่นวาย ช่วยลดความซับซ้อนของโค้ดได้มาก

### Prisma ORM (หรือ Drizzle ORM)

เหตุผล: เป็นเครื่องมือช่วยเขียนคำสั่งคุยกับฐานข้อมูล (SQL) ด้วยภาษา TypeScript ทำให้มี Auto-complete และป้องกันการพิมพ์ชื่อฟิลด์ผิด

---

## 3. ตัวเลือกอื่น (Alternatives)

### Firebase (NoSQL)

(ข้อดี: เริ่มต้นง่ายมาก / ข้อเสีย: เมื่อแอปของคุณต้องการออกรายงานที่ซับซ้อน เช่น รวมยอดรายเดือนแยกตามแพลตฟอร์ม ฐานข้อมูลแบบ Relational อย่าง PostgreSQL ของ Supabase จะทำงานได้ดีและเร็วกว่ามาก)

### เขียน Backend เอง (Node.js/Express)

(ข้อดี: ปรับแต่งได้ 100% / ข้อเสีย: ใช้เวลาพัฒนานาน ต้องดูแลเรื่องความปลอดภัย (Security) และการ Deploy เอง ไม่เหมาะกับคนทำโปรเจกต์คนเดียวที่ต้องการความเร็ว)

---

## 4. ลำดับขั้นตอนการทำงาน (Step-by-step)

- Setup & Auth: สร้างโปรเจกต์บน Supabase และเปิดใช้งานระบบ Login (แนะนำ Google OAuth เพราะผู้ใช้กดคลิกเดียวเข้าได้เลย)

- Database Migration: สร้างตารางข้อมูล (Tables) ผ่าน Prisma หรือระบบของ Supabase

- RLS (Row Level Security): ตั้งค่าความปลอดภัยใน Supabase เพื่อให้ "คนขับ A มองเห็นและแก้ไขได้เฉพาะข้อมูลของคนขับ A เท่านั้น"

- Data Fetching & Mutations: เขียนฟังก์ชันเพื่อดึงข้อมูล (Fetch) มาแสดงใน Dashboard และฟังก์ชันบันทึก/แก้ไขข้อมูล (Mutate)

- Integration: นำฟังก์ชันเหล่านี้ไปเสียบแทนที่ Mock Data ใน Phase 2

---

## 5. สิ่งที่ควรเตรียมก่อนเริ่ม และสิ่งที่ควรทำหลังจบ

### ก่อนเริ่ม

ร่างโครงสร้างฐานข้อมูล (Schema) ในกระดาษให้ชัวร์ก่อนว่าต้องเก็บฟิลด์อะไรบ้าง เพื่อลดการแก้โครงสร้างบ่อยๆ

### หลังจบ

สร้าง "ข้อมูลทดสอบ" (Dummy Data) ปริมาณมากๆ (เช่น ข้อมูลย้อนหลัง 3 เดือน) เพื่อทดสอบว่าหน้า Dashboard สรุปผลโหลดช้าหรือไม่

## 6. โครงสร้างโปรเจกต์ที่แนะนำ (ส่วนที่เพิ่มเข้ามา)

```
src/
 ├── ... (โฟลเดอร์เดิมจาก Phase 2)
 ├── actions/        # Next.js Server Actions (ฟังก์ชันฝั่ง Server)
 │    ├── auth.ts    # จัดการ Login/Logout
 │    └── record.ts  # จัดการบันทึกรายรับ-รายจ่าย
 ├── lib/
 │    └── db.ts      # ตั้งค่าการเชื่อมต่อฐานข้อมูล (Prisma/Supabase)
```

## 7. แนวทางการออกแบบฐานข้อมูล (Database Schema)

เน้นความเรียบง่ายแต่ครอบคลุม (Relational Database):

### users Table

id (PK), email, name, created_at

### platforms Table

id (PK), name (Grab, Bolt, LINE MAN), theme_color

### records Table

id (PK), user_id (FK), platform_id (FK), type (income/expense), amount (Decimal), record_date (Timestamp), note (Text)

## 8. แนวทางการออกแบบ API / การเชื่อมต่อข้อมูล

เราจะใช้ Server Actions เป็นหลัก แทนการทำ REST API แบบดั้งเดิม

ตัวอย่าง: เมื่อผู้ใช้กดปุ่ม "บันทึก" ฟอร์มจะเรียกใช้ Server Action ชื่อ createRecord(data) ซึ่งฟังก์ชันนี้จะวิ่งไปตรวจสอบความถูกต้อง ฝั่ง Server และบันทึกลง Database ทันที ช่วยลดภาระการจัดการ State บน Frontend

---

## 9. Best Practices ที่ควรใช้

- Database Indexing: ทำ Index ที่ฟิลด์ user_id และ record_date ในตาราง records เพราะแอปประเภทนี้จะดึงข้อมูลตาม "ผู้ใช้" และ "ช่วงเวลา" (เช่น ดูสรุปเดือนนี้) ตลอดเวลา การทำ Index จะทำให้ฐานข้อมูลค้นหาได้เร็วปรี๊ด

- Optimistic UI: (ใช้ร่วมกับ Frontend) เมื่อผู้ใช้กดบันทึก ให้ UI อัปเดตข้อมูลบนหน้าจอทันทีโดยไม่ต้องรอเซิร์ฟเวอร์ตอบกลับ (ถ้าเซิร์ฟเวอร์พังค่อยแจ้งเตือนและดึงข้อมูลกลับ) ทำให้แอปดูรวดเร็วมาก

---

## 10. ข้อผิดพลาดที่มือใหม่มักพบ พร้อมวิธีหลีกเลี่ยง

### ข้อผิดพลาด

ดึงข้อมูลดิบทั้งหมดมาบวกเลขหาผลรวม (Sum) บน Frontend (เปลืองเน็ตและเครื่องผู้ใช้ทำงานหนัก)

### วิธีแก้

ให้ Database เป็นคนบวกเลขให้ (ใช้คำสั่ง SQL SUM()) แล้วส่งมาแค่ผลลัพธ์สุดท้าย

### ข้อผิดพลาด

ลืมเปิดใช้งาน RLS (Row Level Security) ทำให้มีช่องโหว่ที่ผู้ใช้คนอื่นอาจใช้ API ดึงข้อมูลของคนอื่นได้

## 11. Deliverables (สิ่งที่ใช้งานได้จริงหลังจบ Phase 3)

- สามารถลงทะเบียนและล็อกอินเข้าใช้งานแอปได้จริง

- เมื่อกรอกข้อมูลและกดบันทึก ข้อมูลจะวิ่งเข้าไปเก็บในฐานข้อมูล

- เมื่อ Refresh หน้าเว็บ หรือไปเปิดในเครื่องอื่น ข้อมูลก็ยังคงอยู่ (Data Persistence)

- Dashboard สามารถคำนวณและแสดงสรุปผลรายรับ-รายจ่ายได้ถูกต้องตามข้อมูลที่มี

# 📱 Phase 4: ทำให้เป็น Progressive Web App (PWA)

## 1. เป้าหมายของ Phase นี้

ทำให้ผู้ใช้สามารถกด "Add to Home Screen" เพื่อติดตั้งแอปได้

ซ่อนแถบ URL ของเบราว์เซอร์ เพื่อให้ UI ดูเป็นแอปพลิเคชันเต็มตัว

ทำให้แอปยังเปิดขึ้นมาได้ (และแสดงหน้า Offline หรือแคชข้อมูลไว้) แม้ในตอนที่ผู้ขับรถขับผ่านจุดที่ไม่มีสัญญาณอินเทอร์เน็ต

---

## 2. เทคโนโลยีและเครื่องมือที่ควรใช้ (The 2026 Choice)

### Serwist (หรือไลบรารี PWA ที่รองรับ Next.js 15+)

เหตุผล: Serwist คือตัวตายตัวแทนของ next-pwa และ Workbox ที่ทันสมัยที่สุดในปัจจุบัน จัดการเรื่อง Service Worker (สคริปต์ที่รันเบื้องหลังเพื่อทำ Caching และโหมด Offline) ได้ง่ายมากเมื่อใช้คู่กับ Next.js

### Web App Manifest (manifest.json)

เหตุผล: ไฟล์พื้นฐานที่บังคับต้องมี เพื่อบอกเบราว์เซอร์ว่าแอปเราชื่ออะไร ใช้ไอคอนแบบไหน และมีสีธีมหลักคือสีอะไร

---

## 3. ตัวเลือกอื่น (Alternatives)

### React Native / Expo

(ข้อดี: ได้แอปแท้ๆ ลง Store ได้ ประสิทธิภาพสูงสุด / ข้อเสีย: ต้องเรียนรู้เฟรมเวิร์กใหม่ จัดการยากกว่าสำหรับคนเดียว และขั้นตอนการเอาขึ้น Store ยุ่งยาก/มีค่าใช้จ่าย)

### Tauri / Capacitor: (แปลงเว็บเป็นแอป)

(ข้อดี: ลง Store ได้ / ข้อเสีย: เพิ่มความซับซ้อนในการตั้งค่าและ Build)

---

## 4. ลำดับขั้นตอนการทำงาน (Step-by-step)

- Prepare Assets: เตรียมไฟล์โลโก้แอป (Icon) ขนาดต่างๆ (เช่น 192x192, 512x512) และทำรูป Splash Screen (รูปตอนโหลดเข้าแอป)

- Create Manifest: สร้างไฟล์ manifest.json ระบุข้อมูลพื้นฐาน เช่น display: "standalone" เพื่อซ่อนแถบเบราว์เซอร์

- Setup Service Worker: ติดตั้งและตั้งค่า Serwist ในไฟล์ next.config.js เพื่อให้ระบบสร้าง Service Worker อัตโนมัติ

- Caching Strategy: กำหนดว่าไฟล์ไหนควรแคชเก็บไว้ในเครื่อง (เช่น ฟอนต์, รูปภาพ, ไฟล์ UI) และข้อมูลไหนต้องดึงใหม่เสมอ (เช่น ยอดเงินอัปเดตล่าสุด)

- Offline Fallback: สร้างหน้า offline.tsx เพื่อแสดงผลเวลาผู้ใช้เน็ตหลุด แทนที่จะปล่อยให้ขึ้นหน้าจอไดโนเสาร์ของ Google Chrome

---

## 5. สิ่งที่ควรเตรียมก่อนเริ่ม และสิ่งที่ควรทำหลังจบ

### ก่อนเริ่ม

เตรียมไอคอนแอปให้สวยงาม เพราะนี่คือสิ่งที่จะอยู่บนหน้าจอมือถือของผู้ใช้ทุกวัน

### หลังจบ

ต้องทดสอบบนมือถือจริงทั้ง iOS (Safari) และ Android (Chrome) เสมอ เพราะพฤติกรรม PWA บนสองระบบนี้มีความต่างกันพอสมควร (เช่น iOS อาจจะต้องกด Share -> Add to Home Screen เอง)

---

## 6. โครงสร้างโปรเจกต์ที่แนะนำ (ส่วนที่เพิ่มเข้ามา)

```

 ├── icons/          # เก็บไฟล์รูปภาพโลโก้ขนาดต่างๆ
 ├── manifest.json   # ไฟล์กำหนดคุณสมบัติแอป
 └── sw.js           # Service Worker (สร้างอัตโนมัติโดยไลบรารี)
```

## 7. แนวทางการออกแบบฐานข้อมูล (ไม่มีการเปลี่ยนแปลง)

แต่ในฝั่ง Frontend อาจจะมีการใช้ IndexedDB หรือ LocalStorage เพิ่มเติมเพื่อเก็บข้อมูลชั่วคราวเวลาออฟไลน์

---

## 8. แนวทางการออกแบบ API / การเชื่อมต่อข้อมูล (Offline First - ทางเลือกเสริม)

หากเน็ตหลุดตอนกด "บันทึก" เราสามารถเก็บ Transaction นั้นไว้ใน LocalStorage ก่อน พอระบบตรวจจับได้ว่าอินเทอร์เน็ตกลับมา ค่อยยิง API (Server Action) ไปบันทึกที่ Supabase (เรียกว่า Background Sync)

## 9. Best Practices ที่ควรใช้

- App Shell Architecture: แคชโครงสร้างหลักของแอป (Navbar, เมนูด้านล่าง) ไว้เสมอ เพื่อให้กดเข้าแอปปุ๊บ หน้าจอขึ้นปั๊บ (รู้สึกเร็ว) แล้วค่อยขึ้น Loading หมุนๆ เพื่อดึงข้อมูลทีหลัง

- Custom Install Prompt: เขียนโค้ดดักจับและสร้างปุ่ม "ติดตั้งแอปของเราสิ" แบบสวยๆ ภายในแอป แทนที่จะรอให้เบราว์เซอร์เด้งเตือนแบบทื่อๆ

## 10. ข้อผิดพลาดที่มือใหม่มักพบ พร้อมวิธีหลีกเลี่ยง

### ข้อผิดพลาด

ตั้งค่า Caching ผิดพลาด ทำให้ผู้ใช้เห็นแอปเวอร์ชันเก่าตลอดเวลา แม้เราจะอัปเดตโค้ดใหม่ไปแล้ว (Cache นรก)

### วิธีแก้

ศึกษากลยุทธ์ Network First (ขอดูข้อมูลใหม่จากเน็ตก่อน ถ้าไม่มีค่อยเอาแคชมาโชว์) และจัดการ Cache Invalidation ให้ดี

### ข้อผิดพลาด

ละเลยผู้ใช้ iOS

### วิธีแก้

ต้องเข้าใจว่า Apple/iOS สนับสนุน PWA ช้ากว่า Android ต้องทดสอบให้มั่นใจว่าฟีเจอร์หลักทำงานได้บน Safari
